# 基于 WebGL 与 Three.js 的三维跑酷游戏设计与实现

## 1. 项目概述

### 1.1 项目名称

霓虹金币跑酷。

### 1.2 项目类型

本课程设计选择作品设计型题目，作品形式为网页三维游戏。项目已经完成浏览器端运行，并通过 Electron 打包为 Windows 免安装可执行程序，可以作为课程设计作品进行提交和演示。

### 1.3 项目背景与简介

多媒体技术课程不仅涉及图像、音频、动画等内容，也包含三维图形显示、人机交互和多种媒体资源的综合应用。基于这一课程特点，本设计没有选择单纯的视频剪辑或理论论文，而是实现了一个可以实际运行的三维跑酷小游戏。游戏以悬浮赛道为主要场景，玩家控制一个霓虹小球在三条赛道之间移动，通过跳跃、换道等操作躲避障碍物，同时收集道路上的金币来获得分数。

游戏的核心技术路线是 WebGL 与 Three.js。Three.js 负责三维场景、模型、灯光、相机、材质和动画渲染，JavaScript 负责游戏状态、玩家输入、关卡生成、碰撞检测和分数计算。为了便于提交，项目又使用 Electron 将网页游戏打包为 Windows 桌面程序，使其既可以在浏览器中运行，也可以通过 `.exe` 文件直接打开。

### 1.5 设计目标

本设计的目标是完成一个规则清晰、画面完整、能够交互操作的三维跑酷小游戏。作品需要体现三维场景渲染、模型加载、动画表现、键盘交互、音效反馈和桌面打包等内容，同时也要保证项目结构清楚，能够通过源代码复现运行。与只展示静态三维模型相比，跑酷游戏更能体现动态交互和实时渲染；与理论论文相比，作品设计也更适合展示多媒体技术的实际应用效果。

## 2. 需求分析

### 2.1 功能需求

根据课程作品设计的要求和游戏本身的玩法，本系统需要具备玩家控制、三维场景显示、金币收集、障碍躲避、碰撞检测、分数统计、速度变化、游戏状态切换和桌面打包等功能。玩家在游戏中能够通过键盘控制小球左右移动和跳跃，系统需要根据玩家位置判断是否收集金币、踩到跳板或撞到障碍物。游戏运行过程中速度会逐渐提高，使后期操作难度增加。界面上需要显示当前分数、金币数量、最高分和前进距离，失败后可以重新开始。

| 功能类别 | 具体说明 |
|---|---|
| 玩家控制 | 支持左右换道、跳跃、暂停、继续和重新开始 |
| 场景渲染 | 显示三维赛道、城市建筑、星空背景、灯光、雾效和粒子效果 |
| 金币系统 | 在赛道上生成金币，玩家碰到金币后增加金币数和分数 |
| 障碍系统 | 生成高障碍、低障碍、跳跃障碍、门框障碍和光门装饰 |
| 碰撞检测 | 判断玩家与金币、跳板、障碍物之间的接触关系 |
| 游戏状态 | 支持准备、运行、暂停、失败和重新开始等状态 |
| 模型加载 | 加载 Kenney 模型资源，模型不可用时自动使用基础几何体兜底 |
| 桌面打包 | 使用 Electron Builder 生成 Windows 免安装版程序 |

### 2.2 非功能需求

除了基本功能外，游戏还需要保证运行稳定和提交方便。首先，玩家输入应当有较快响应，左右换道和跳跃动作不能明显卡顿。其次，画面中的障碍物与实际碰撞判定要尽量一致，避免玩家看到空路却被判定失败的情况。再次，模型资源加载失败时不应导致游戏崩溃，而应显示可替代的基础模型。项目还需要具备清晰的目录结构和可复现的运行方式，使教师或同学可以通过 `npm install`、`npm run dev`、`npm run build` 等命令检查源代码运行情况。

### 2.3 用户操作方式

玩家主要通过键盘进行操作。`A` 或左方向键用于向左换道，`D` 或右方向键用于向右换道，空格键、`W` 或上方向键用于跳跃。游戏过程中按 `P` 可以暂停或继续，按 `R` 可以重新开始。这样的按键设计接近常见跑酷游戏操作方式，学习成本较低，适合课堂展示时快速上手。

## 3. 系统总体设计

### 3.1 技术路线

本设计采用 Vite、Three.js、JavaScript 和 Electron 组合实现。Vite 用于前端开发和生产构建，Three.js 用于 WebGL 三维场景渲染，JavaScript 用于游戏逻辑和交互控制，Vitest 用于核心逻辑测试，Electron 用于把网页游戏打包为桌面应用。选择这一技术路线的原因是它不需要 Unity 等大型游戏引擎，项目体积相对较小，也更符合“网页三维游戏”的作品形式。

### 3.2 系统总体架构

![系统总体架构图](images/system-architecture.png)

从总体结构看，系统可以分为输入层、运行窗口层、游戏主逻辑层、渲染层、游戏规则层、模型资源层和打包发布层。用户通过键盘输入操作，浏览器或 Electron 窗口负责承载 Canvas 和页面 UI。`src/main.js` 是游戏运行的中心文件，负责场景初始化、动画循环、关卡片段生成和对象更新。状态管理、碰撞检测和模型兜底逻辑分别拆分到 `src/game/` 目录下，便于测试和维护。模型资源放在 `public/models/` 中，打包时会随静态资源一起进入最终程序。

## 4. 游戏流程设计

### 4.1 游戏主流程

游戏启动后首先进入准备状态，页面中央显示开始面板。玩家点击开始后，系统初始化分数、金币、速度、玩家位置和关卡对象，然后进入运行状态。在运行状态下，主循环会持续读取玩家输入，更新小球的横向位置和跳跃高度，同时推动道路上的金币、障碍物、跳板和装饰物向玩家方向移动。当玩家碰到金币时增加分数，踩到跳板时触发更高跳跃；如果碰到不可通过的障碍物，游戏状态切换为结束，并显示最终分数和重新开始按钮。

![游戏主循环流程图](images/game-loop.png)

图 2 游戏主循环流程图。

### 4.2 游戏状态设计

游戏状态主要记录当前运行阶段、玩家所在车道、目标车道、分数、金币数、最高分、速度、运行时间、玩家高度、垂直速度和是否落地等数据。其中 `status` 用于区分 ready、running、paused 和 ended。`laneIndex` 和 `targetLaneIndex` 用于描述玩家当前位置和目标车道，配合插值算法实现平滑换道。跳跃部分则通过 `playerY`、`verticalVelocity` 和 `isGrounded` 来模拟起跳、上升、下落和落地过程。

这种状态设计的好处是游戏逻辑比较集中，开始、暂停、失败和重新开始都可以通过状态切换完成。游戏失败后不会直接刷新页面，而是重置状态和场景对象，这样玩家可以在同一窗口中继续尝试。

## 5. 功能模块设计

### 5.1 三维场景模块

三维场景由 Three.js 创建，主要对象包括透视相机、WebGL 渲染器、环境光、方向光、点光源、玩家小球、悬浮赛道、金币、障碍物、城市建筑和粒子背景。相机设置在玩家后上方，视线朝向赛道前方，使画面能够体现前进方向上的空间纵深。场景中还加入了雾效和星空背景，用来增强距离感和三维氛围。

【截图：三维赛道和城市建筑背景】

### 5.2 玩家控制模块

玩家小球可以在三条赛道之间移动。换道不是瞬间改变坐标，而是先记录目标车道，再通过阻尼插值逐渐移动到目标位置。这样可以避免画面突兀，也能让玩家感觉角色是在真实移动。跳跃部分采用简化的重力模型：按下跳跃键后给小球一个向上的初始速度，随后在重力影响下逐渐下降，落回赛道后恢复可跳跃状态。

### 5.3 关卡片段生成模块

地图并不是完全随机生成，而是由若干固定片段循环组合。这样做主要是为了保证游戏节奏和难度可控，避免出现完全无法通过或视觉上不合理的组合。当前关卡中包含金币串、低障碍、跳跃障碍、跳板、门框障碍、光门装饰和空中金币等内容。部分外观模型会在箱子、尖刺、栅栏等模型之间随机选择，城市两侧建筑也会随机变化，因此每次游玩时画面不会完全相同。

### 5.4 碰撞检测模块

碰撞检测采用 AABB 轴对齐包围盒方式实现。系统为玩家和可交互对象分别建立简化碰撞盒，并判断它们在 x、y、z 三个方向上的范围是否重叠。如果三个方向都发生重叠，就认为发生碰撞。金币碰撞会触发收集效果，跳板碰撞会触发更高跳跃，障碍物碰撞则根据障碍类型和玩家高度判断是否失败。

开发过程中曾出现过视觉位置和碰撞判定不一致的问题。为了解决这类误判，系统对碰撞逻辑做了几个限制：光门和纯装饰物不参与碰撞；单车道障碍只与玩家当前可见车道进行比较；低障碍在玩家跳跃高度足够时可以通过；高障碍必须通过换道躲避。这样处理后，碰撞结果与玩家在画面中看到的位置更加一致。

![碰撞检测与误判修复流程图](images/collision-flow.png)

### 5.5 模型资源与兜底模块

项目使用了 Kenney 的免费 3D 模型资源，主要用于城市建筑、金币、箱子、尖刺、栅栏和平台等对象。模型文件放在 `public/models/kenney/` 目录下，运行时通过 Three.js 的 `GLTFLoader` 加载。

在实际调试中发现，外部模型可能因为路径、材质、层级或可见性问题而显示异常。如果仍然给这些不可见模型保留碰撞体，就会出现“空气碰撞”。因此项目增加了模型可见性检查，只有模型中存在可见 mesh，且父节点和材质都处于可见状态时，才会使用该模型。否则系统会自动使用 Three.js 基础几何体作为替代模型。对于高障碍，还额外增加了醒目的可见标记，保证危险位置在画面中能够被玩家看到。

游戏界面上方显示 Score、Coins、Best 和 Progress，用来展示当前分数、金币数、最高分和前进距离。中央面板用于显示开始、暂停、失败和重新开始等状态。玩家收集金币、触发跳板或游戏失败时，画面会产生粒子反馈，并配合音效增强交互感。UI 的设计尽量保持简洁，避免遮挡赛道和障碍物。

## 6. 关键技术实现

### 6.1 Three.js 场景构建

Three.js 场景由 `Scene`、`PerspectiveCamera` 和 `WebGLRenderer` 组成。渲染器绑定到 HTML 页面中的 Canvas，负责把三维对象绘制到屏幕上。场景中使用环境光提供整体亮度，使用方向光和点光源强化物体明暗变化。赛道、金币、障碍物和建筑都以三维对象形式加入场景，配合相机透视和雾效形成三维空间效果。

### 6.2 玩家移动与跳跃

玩家左右移动采用目标车道加平滑插值的方式实现。当玩家按下左右方向键时，系统只修改目标车道，真实坐标会在每一帧逐渐靠近目标坐标。跳跃使用垂直速度和重力模拟，按键触发后小球获得向上速度，随后在重力作用下回落。这个实现不需要完整物理引擎，但能够满足跑酷游戏中跳跃障碍和踩跳板的需求。

### 6.3 关卡片段循环

关卡生成采用固定片段循环加外观随机的方式。固定片段保证道路组合有基本规律，例如跳板前方会有需要跳跃的障碍，金币会分布在可通过的位置；外观随机则让障碍物、建筑和装饰表现不完全重复。随着游戏时间增加，物体移动速度逐渐提高，因此后期难度会自然上升。

### 6.4 AABB 碰撞检测

AABB 碰撞检测的计算过程比较简单，只需要比较两个包围盒在 x、y、z 三个方向上的最小值和最大值是否重叠。跑酷游戏中的金币、障碍物和跳板数量较多，如果使用复杂物理碰撞会增加计算开销。AABB 虽然不适合特别复杂的模型碰撞，但对于本游戏这种规则化赛道和简单障碍已经足够。

### 6.5 车道过滤与误判修复

三车道跑酷中比较容易出现一个问题：玩家已经按键换道，但画面中的小球还没有完全移动到目标车道。如果此时直接使用目标车道进行碰撞判断，就可能出现视觉上没有碰到障碍却被判定失败的情况。为了解决这个问题，系统根据玩家当前可见 x 坐标计算最近车道，再用于单车道障碍过滤。这样碰撞判断更接近玩家在屏幕上看到的位置。

### 6.6 模型加载失败处理

模型资源虽然能提升画面效果，但也会带来加载失败或显示异常的风险。本设计在模型加载后会检查模型内部是否有可见 mesh，如果模型不可用，就创建箱体、圆柱、圆锥等基础几何体作为替代对象。这个机制保证了游戏在资源异常时仍能正常运行，也避免了不可见障碍造成的误判。

### 6.7 Electron 打包

网页端完成后，项目使用 Electron 进行桌面打包。Vite 先将网页资源构建到 `dist/` 目录，Electron 主进程再加载构建后的 `index.html`。最后通过 Electron Builder 生成 Windows 免安装程序。这样提交时既可以提供源码，也可以提供可直接双击运行的程序。

## 10. 项目特色与创新点

本设计的主要特点在于使用网页技术完成三维游戏，而不是依赖 Unity 等大型引擎。游戏通过 Three.js 实现三维赛道、模型、灯光和动画，再通过 Electron 打包为桌面程序，兼顾了网页展示和可执行文件提交两种需求。关卡生成没有采用完全随机方式，而是使用固定片段和随机外观结合的方法，使游戏既有可控的通过逻辑，也有一定视觉变化。针对开发中出现的空气碰撞问题，项目增加了模型可见性检测、几何体兜底和车道过滤，使视觉表现与碰撞判定更加一致。核心状态、碰撞和模型辅助逻辑还配有自动化测试，便于后续继续修改。

## 11. 遇到的问题与解决方法

### 11.1 障碍物视觉与碰撞不一致

早期版本中，玩家有时会感觉自己没有碰到障碍物，但游戏却判定失败。经过排查，原因主要有三个方面：部分模型实际尺寸和视觉尺寸不完全一致，某些模型加载异常后不可见，玩家换道过程中目标车道和真实位置之间存在短暂差异。后续通过模型可见性检查、基础几何体兜底、高障碍可见标记和当前可见车道判断等方式进行修复。修复后，碰撞判断更接近玩家实际看到的位置。

### 11.2 光门与障碍物语义不清晰

最初的光门设计容易让玩家误以为需要躲避或跳跃，但画面上又不像普通障碍物，导致规则理解不清。后续将光门统一改为装饰物，不再参与碰撞，真正需要避开的对象则使用实体障碍表现。这样处理后，玩家能够更直观地区分装饰效果和危险障碍。

### 11.3 Electron 打包路径问题

项目在中文目录下打包时，Electron Builder 可能因为输出目录重命名而出现权限或路径问题。为了解决这个问题，打包脚本先把产物输出到英文临时目录 `C:\runner-release`，然后再复制到项目的 `release/` 目录。这样既保留了原项目位置，也提高了打包成功率。
